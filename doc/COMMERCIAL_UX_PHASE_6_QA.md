## 1. 结论

消费者不再被直接送入单一影片/影院/日期的选座页。页面现在先提供 3 部影片、2 家影院和未来 3 个营业日，随后展示该组合的具体场次、票种与库存。24 个场次拥有独立确定性库存，订单快照继续记录实际影片、影院、影厅和时间。

可售判断由 `CommercialBookingService` 统一输出；目录 UI 不再复制开售/停售时间规则。选择已停售影片组合时，系统优先保留影院并跳到最近可售日期；日期切换若会迫使影片或影院暗中变化，则该日期不可选。

## 2. Before / After / Why

| Before | After | Why |
| --- | --- | --- |
| 页面启动即固定“星际回响 / 光影中心 / 当天” | 先选择电影、影院、日期，再选择具体时间 | 真实购票决策先确定内容与地点，座位只属于具体场次 |
| UI 通过 `bookingClosesAt` 自行判断停售 | 应用层返回统一 `availability` 视图 | 开售规则是业务事实，不能散落在 DOM controller |
| 当天场次过期后只能进入下一营业日的单一目录 | 每个影片和影院可落到最近可售日期 | 保持用户选择意图，同时避免展示没有下一步的死组合 |
| 切换场次只播报时间 | 播报影片、影院、日期与时间，并归还触发按钮焦点 | 上下文变化范围扩大后，需要完整反馈和稳定键盘位置 |
| 海报图形与文字写死为一部影片 | 海报语义、标题与克制的三套图形主题随影片更新 | 视觉变化服务于内容识别，不引入外部图片依赖和装饰动画 |
| 单场次集合只有 4 份库存 | 24 个场次各自初始化、查询和运维 | 跨电影/影院/日期绝不能共享售出或锁座状态 |

## 3. 响应式与交互证据

- 1440px 下三组选择器同屏，影片、影院与日期层级清楚；
- 390px 下影片和影院选择器只在自身容器横向滚动，页面 `scrollWidth === clientWidth`；
- 所有目录按钮高度 54px；重绘后焦点回到新的同义触发按钮；
- 电影切换会清除尚未锁定的座位并保留票种和推荐偏好；有效锁座存在时禁止切换上下文；
- 高频选座与目录列表没有入场动画，按钮只使用 140ms 按下反馈；
- 不可售日期使用原生 disabled 语义，具体场次显示“已停售”而非只降低透明度。
