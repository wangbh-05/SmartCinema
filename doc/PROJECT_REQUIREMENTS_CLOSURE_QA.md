# SmartCinema 本地需求闭环 QA

> 日期：2026-07-18
> 分支：`zcjx/smart_cinema`
> 状态：通过
> 范围：本地项目需求、核心交易闭环、前端模块与辅助体验；不包含真实支付、渠道退款或二维码验票。

## 1. 结论

本轮不是继续堆叠商业影院后台或支付能力，而是修正此前重构中“删除旧 UI 时同时删除需求能力”的偏差。当前产品继续使用清晰的商业购票漏斗，但已经以新的领域模型和交互位置覆盖本地项目所需的三种影厅、人群/同行推荐、热度参考、四维座位体验、无障碍偏好和完整订单闭环。

登录/注册 Dialog 的原始缺陷继续由真实浏览器契约保护：存在固定关闭键；背景点击和内容拖出均不关闭；Escape 只关闭最上层；关闭后焦点归还触发点。

## 2. Before / After / Why

| Before | After | Why |
| --- | --- | --- |
| 删除旧 Canvas 后只剩一个 180 座影厅模板 | 场次分别引用 100、200、300 座三种目录影厅，座位图按目录动态渲染 | 三厅应是业务目录事实，不应回到用户可随意修改的技术开关 |
| 推荐只按票数和位置偏好，不再覆盖原人群/同行规则 | BookingDraft 保存同行方式；儿童/长者约束来自票种；5–20 人团体可在大厅同排连续 | 恢复本地需求，同时不重新收集姓名、年龄列表等无必要信息 |
| 热度与评分随旧功能台一起被删除 | 同一座位图可按需显示形状化热度；选座后显示视野、距离、周边空位、价格四维决策提示 | 能力应服务当前决策，不应以第二张画布或购前主观评价争夺注意力 |
| “增强可读性”一个开关含义模糊 | 大字体、高对比、色觉友好、减少动态四项独立设置 | 每个设置对应明确需求、样式和持久化契约，用户能预期结果 |
| `commercial.js` 继续吸收新模块渲染 | 新增 `CommercialDecisionSupportController`；领域计算位于 `SeatDecisionGuide` | 防止重建旧的万能入口脚本，保持 UI → Application → Domain 边界 |
| 右侧摘要声明 sticky，但父列与卡片同高，滚动后实际离开视口 | 摘要列随左侧流程拉伸，并增加滚动几何回归 | 主要金额与 CTA 必须在桌面选座过程持续可见 |
| 大厅横向滚动时固定排号会被裁切座位文字穿透 | 固定排号增加舞台色遮罩与层级 | 横向浏览时仍能可靠判断排号，不用依赖滚动位置记忆 |
| 路线图继续把真实支付、二维码和完整账户列为待办 | 明确它们不属于本地项目范围；文字取票码、演示确认和政策退票构成闭环 | 避免为不存在的外部渠道建立虚假复杂度，也避免文档过度承诺 |

## 3. 结构与不变量

- `BookingDraft` 的绝对票数上限为 20；9–20 张只允许“多人同行”；
- 单人必须 1 张、双人必须 2 张、家庭为 2–8 张、多人为 5–20 张；
- 推荐的儿童票座位不在前三排，长者票座位不在后三排；两者同时出现时只使用中间四排；
- 团体推荐保持同排、同区、连续，不跨过道拼接；30 座大厅中央区可容纳 20 人；
- 热度由影厅结构和当前场次库存派生，不持久化第二份事实；
- 四维体验由当前所选座位、库存和价格政策派生，不进入订单或存储；
- 交易价格仍只由 `PricingQuote` 计算；决策提示不参与收费；
- UI 不直接访问 Web Storage，新增 controller 只通过应用服务取得派生视图。

## 6. 视觉与辅助验收

- 390×844：大厅、移动底栏、同行方式、热度开关和座位舞台无页面级横向溢出；
- 1440×1000：大厅推荐后 summary 固定在 `top=88px`，座位图内部横向滚动；
- 大字体 + 高对比：Dialog 仍完整容纳四项设置，边框、焦点和主要 CTA 清楚；
- 色觉友好：已售席使用斜纹，选中席使用内描边，优选席使用虚线，不仅依赖颜色；
- 热度参考：热门使用实心圆、一般使用菱形、冷门使用空心圆，并提供文字图例；默认关闭；
- 高频座位交互没有入场位移动画；reduced-motion 继续将过渡降为即时反馈。

## 7. 明确边界

真实支付、支付渠道退款、二维码验票、完整个人资料、卖品和会员营销不进入本地项目闭环。真实读屏需要在目标操作系统与读屏软件上人工验收；当前自动化覆盖逐座名称/状态、单一 Tab 停点、方向键、Space、焦点保持和 `aria-live` 反馈。
