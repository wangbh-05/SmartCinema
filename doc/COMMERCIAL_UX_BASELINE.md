## 1. 基线结论

当前版本是一个可靠、可测试的影院座位功能演示，但不是完整的商业购票产品。

- 技术正确性基线良好：Node 测试 105/105；上一阶段浏览器契约 12/12；
- 信息架构基线不合格：购买对象、购买顺序和交易状态不完整；
- 视觉实现本身不存在重大溢出或弹窗缺陷，但视觉层级忠实呈现了错误的产品结构；
- 本轮不以“修 CSS”开局，先补齐产品领域和交易状态机。

## 3. 真实浏览器基线

### 1440 × 1000

- 首屏同时呈现选座、智能推荐、观影体验、订单四类任务；
- 下方立即出现第二张近似座位图“影院热度地图”；
- 购买对象只显示“中厅 · 200 座 / 周四”，没有电影、影院、具体时间、制式、语言、票种或退改信息；

- 订单 CTA 与推荐/评分同级，没有形成唯一主操作。

## 4. 代码事实

| 事实 | 当前证据 | 影响 |
| --- | --- | --- |
| Showtime 只编码影厅类型与星期 | `src/domain/cinema/Showtime.js:4-41` | 无法表达电影、影院、开场时间、制式和销售状态 |
| 进入结算前强制登录 | `src/application/booking/StartCheckout.js:12-16` | 身份摩擦发生在锁座价值建立之前 |
| CheckoutIntent 只检查 sold 后写 SessionStorage | `src/application/booking/StartCheckout.js:17-49` | 座位没有进入可被其他请求看见的有效 hold |
| 售出只发生在确认订单 | `src/domain/order/BookingPolicy.js:5-15` | 倒计时期间另一个用户仍可选择相同座位 |
| 首页直接操作技术影厅与星期 | `index.html:24-45` | UI 暴露内部模型而非用户购买对象 |
| 推荐需要年龄、关系和成员姓名 | `index.html:87-126` | 增加无必要 PII 与认知负担 |
| 评分、订单、热图、设置同页 | `index.html:130-199` | 页面按功能模块而非购买依赖组织 |
| 座位价格由位置在 mutable view model 中计算 | `src/core/SeatData.js:139-157` | UI 视图模型成为价格事实源，无法表达票种/费用政策 |
| 可用性、推荐、热度与选择共存在 SeatData | `src/core/SeatData.js` | 交易事实与可视派生状态仍有遗留耦合 |

## 5. 必须保持的成果

- `domain → application → infrastructure/ui` 的依赖方向；
- revision 驱动的持久化冲突保护；
- 稳定 `userId` 的订单授权；
- 幂等确认订单；
- 远端 hold 与本地选择隔离；
- 统一 Dialog、焦点归还、Escape 和遮罩拖动边界；
- 输入快捷键隔离；
- 320px 无页面横向溢出；
- 大字体、色盲和 reduced-motion；
- 安全 DOM 文本渲染与备份回滚。

## 6. 冻结迁移样本

`tests/fixtures/state-v2-commercial-migration.json` 是第二阶段 v2→v3 的固定输入，覆盖：

- 管理员与普通用户；
- 已登录 session；
- 用户设置；
- 一个 `medium:day:3` 库存；
- 一个已确认双座订单；
- v1→v2 migration 元数据。

Storage v3 迁移必须保证：

1. 原始 v2 key 在迁移提交成功前不删除；
2. 用户和订单稳定 ID 不变化；
3. 旧 `showtimeId` 映射到明确的 legacy catalog 场次；
4. 旧订单价格和座位作为不可变快照保留；
5. 无法确定的新字段使用显式 `legacy-*`/`unknown`，不得编造业务事实；
6. 迁移失败返回可读报告并保持 v2 数据可恢复。
