# SmartCinema 商业产品要求与课程作业要求调和清单

> 日期：2026-07-19
> 作者：GrainRain `<zcjx24@mails.tsinghua.edu.cn>`
> 分支：`zcjx/smart_cinema`
> 依据：`doc/AGENT.md`、当前生产代码与自动化测试
> 结论：商业购票演示闭环已经成立，但课程作业的若干显式评分点仍未实现，不能再表述为“全部等价覆盖”。

## 1. 如何阅读本清单

本项目同时面对两套不同的成功标准：

1. **课程作业要求**是验收合同。只要原文明确指定 Canvas、颜色、时间范围或提交格式，采用商业上更合理的替代方案并不自动得分。
2. **商业产品要求**关注交易真实性、可恢复性、无障碍、隐私最小化和操作效率。它可以解释为什么当前设计更合理，但不能覆盖作业硬性条款。

后续实现应优先做到“共享同一领域模型，按展示层适配两套验收”，避免重新维护两套库存、订单和推荐算法。

## 2. 冲突点明细

| 议题 | 课程作业要求 | 真实商业影院通常要求 | 当前实现 | 冲突与风险 | 建议调和方式 |
| --- | --- | --- | --- | --- | --- |
| 座位渲染 | 主界面必须使用 Canvas 绘制三种影厅 | 优先语义化控件、键盘和读屏可操作；Canvas 需要额外无障碍层 | 100/200/300 座均为 DOM `button`，架构测试还明确禁止生产 Canvas | 商业可访问性更好，但严格评分点不满足 | 抽象 `SeatMapRenderer`；Canvas 负责视觉绘制，保留同步的语义座位列表或透明控制层 |
| 座位颜色 | 空座绿、选中黄、已售红 | 颜色需符合品牌和风险语义，且不能只依赖颜色 | 当前为舞台深色体系，状态另有纹理、描边和文字 | 严格色值/语义不一致 | “课程验收模式”使用指定三色，同时保留图形、纹理和文字标签 |
| 登录门槛 | 登录认证后方可操作 | 常允许访客浏览、选票选座和短时锁座，在支付/确认前登录 | 访客可完成浏览、选票、推荐、选座和锁座，确认订单前登录 | 商业转化更顺，但“登录后方可操作”可能被判不满足 | 明确选择：若以得分优先，首次改票/选座前弹出认证；若保留商业漏斗，在报告中说明差异但接受评分风险 |
| 管理员入口 | 管理员登录可进入后台 | 消费者入口不应暴露运维入口，后台使用独立受权 URL | `internal.html` 独立且受角色门控，消费者导航不暴露 | 管理员在消费者页登录后缺少可发现的后台跳转 | 仅对管理员显示“进入运维”链接；普通用户与访客完全不可见 |
| 用户分类 | 成年人、老年人、少年；少年小于 15，老人至少 60 | 票种资格与座位偏好应分离，尽量不采集逐人成员资料 | 儿童票写 12 岁以下；长者票 60+；不收集成员姓名/年龄 | 少年阈值与作业不一致 | 将儿童/少年票阈值统一为 `<15`，或新增少年票；继续采用票种数量而非姓名列表 |
| 情侣/家庭/团体 | 情侣中间连续双座；家庭中后排；5–20 人团体同排且服从年龄规则 | 还要考虑银幕视角、过道、已占座密度、价格区和隐私 | 情侣已按中后排中央、连座和周边空位排序；家庭/团体规则存在 | 当前情侣规则优于最低要求，但阈值仍受票种差异影响 | 保留新评分，并给规则添加独立测试和可解释理由 |
| 手动多选 | 单选；Ctrl+单击多选；推荐后可改座 | 票数确定后通常直接点击切换，不要求用户理解 Ctrl | 点击即按票数多选，可覆盖推荐；不要求 Ctrl | 商业交互更自然，但按字面可能缺少 Ctrl 行为 | 兼容 Ctrl+单击但不把它设为唯一入口；文档说明普通点击也可完成 |
| 拖拽选座 | 加分项 | 拖拽容易误选，尤其在触控滚动和文字选择时 | 未实现；登录 Dialog 已特别修复“拖出误关闭” | 加分未取得，且直接加拖拽可能制造新误触 | 仅在 Canvas 课程模式实现带阈值的框选；触控默认关闭并提供撤销 |
| 热度地图 | Canvas；红/黄/蓝；动态展示一周变化 | 应区分预测热度、真实销量和当前库存，避免伪造实时数据 | DOM 座位图上的按需热度层；由结构和当前库存派生；没有一周时间轴 | 商业提示可用，但 Canvas 和 7 天动态两个评分点未满足 | 以 7 天确定性样本驱动同一热度模型，Canvas 绘制并提供日期切换/播放控制；明确“演示数据” |
| 观影体验评分 | 视角、距离、周围空位；极佳/优秀/一般；观影后人工评分与系统分综合 | 购前推荐分与观后评价属于不同业务阶段，不应混为同一分数 | 购前显示视野、距离、周边空位、价格平衡；等级含极佳/优秀/舒适/基础；无观后评分 | 当前决策辅助更适合购前，但缺少明确作业功能 | 保留购前决策分；在已完成订单中增加观后评分，再以明确权重展示综合分，不反向修改历史订单价格 |
| 语音与读屏 | 可选语音提示 | 优先标准读屏语义、焦点管理和用户系统设置，避免页面抢读 | 有 ARIA、键盘、焦点陷阱和 `aria-live`，没有 Web Speech | 标准无障碍更可靠，但可选加分/展示点缺失 | 将语音作为显式开启的增强项；不取代 ARIA，不默认播放 |
| 一周热度与三日排期 | 热度观察窗为 7 天 | 可售排期窗口可独立为 3 天或更长 | 可售排期为本地当天起 3 天，按三套模板滚动 | 两个“日期窗口”语义不同，不应强行合并 | 保持 3 天售票排期；热度模块单独维护过去/模拟 7 天数据 |
| 实时更新 | WebSocket 模拟为加分项 | 真实系统要求服务端权威库存、连接恢复和冲突解决 | 本地确定性库存与冲突恢复，无 WebSocket | 不伪造网络更诚实，但加分未取得 | 若实现，仅做明确标注的多标签页 `BroadcastChannel` 或本地 WebSocket 演示，不能宣称真实并发 |
| 单文件提交 | 整理为一个可直接打开的 `index.html` | 工程源码应模块化、可测试、可维护，通常经构建后发布 | ES modules、多 CSS/JS 文件、两个入口，需 HTTP 服务 | 当前工程质量高，但不满足交付包字面要求 | 保留模块化源码；新增可复现的导出脚本，生成单文件 `dist/index.html` 作为提交物 |
| 三步流程 | 操作流程尽量不超过 3 步并提交流程图 | 商业交易还要显示票种、价格、锁座、身份和政策确认 | 页面标为“场次 → 票座 → 确认”三阶段，但每阶段含多个动作 | 高层符合，实际动作不可能只有三次点击 | 报告中以三阶段而非三次点击解释；补最终流程图和异常分支 |
| 真实资料 | 作业未要求接入真实影院接口 | 真实名称、地址、排期和价格必须区分来源与实时性 | 影片为真实条目；影院名称、地址、影厅、排期与价格均为本地虚构演示 | 若把虚构影院写成真实门店会形成误导 | 页面常驻披露；影片与封面列明来源，虚构影院不绑定真实商户 |

## 4. 推荐实现顺序

1. **先确定验收策略**：是否严格要求“未登录不能操作”。这一项会改变主漏斗，必须由项目负责人明确。
2. **建立双渲染边界**：领域座位、库存、推荐不动；新增 Canvas/语义混合渲染器，满足三厅、弧形、座号和指定颜色。
3. **补作业数据语义**：少年阈值改为 15；建立独立 7 天热度历史；评分等级提供三档课程映射。
4. **补观后环节**：只有已完成且已过场次的订单可评分，系统分与人工分分别保存并展示综合结果。
5. **补交付链**：生成单文件 `dist/index.html`、最终 `report.pdf`、流程图、参考资料和 AI 协作记录。
6. **最后处理加分项**：拖拽、语音、WebSocket/多标签页模拟、规则型问答顾问；每项都不得破坏当前交易闭环。

## 5. 本轮目录改造与事实边界

- 影片为 7 部指定经典影片：星际穿越、你的名字、怦然心动、闻香识女人、哈利·波特与魔法石、疯狂动物城、楚门的世界。
- 影院名称简化为 3 家采用清华周边地名的虚构演示影院：五道口、清河、中关村。名称仅用于地域语境，道路地址、影厅和商业资料均不对应真实商户。
- 可售窗口始终为本地北京时间当天及随后两天。三套日排期按绝对日期循环，因此窗口前移后，新加入的第 3 天会复用刚结束日期的排期模板。
- 每个营业日提供 36 场、三日共 108 场；每部影片每天至少覆盖 4 场。影片名称和片长是资料字段；影院名称、地址、场次、价格、影厅名称、容量和可售状态均为本地虚构演示。每家影院以两间目录影厅并行排期，300 座影厅首次打开时居中展示。
- 影片封面素材从 TMDB 获取后保存在 `public/images/posters/`，运行时从项目本地加载；本地文件缺失或解码失败时保留项目自有 CSS 抽象图形兜底。封面仅用于课程演示，正式发布前仍需核对图片授权与署名要求。

## 6. 决策记录

当前建议不是恢复一套旧 Canvas 功能台，也不是用“商业上更合理”否定课程要求。最稳妥的方向是：**保留现在的商业交易和领域架构，在 UI 层增加课程验收需要的 Canvas/色彩/7 天/评分/单文件交付能力。** 这样能够同时守住代码质量和评分合同。
