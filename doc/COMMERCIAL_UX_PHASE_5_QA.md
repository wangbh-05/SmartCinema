# SmartCinema Phase 5：内部运维与旧链退出 QA

> 日期：2026-07-18  
> 分支：`zcjx/smart_cinema`  
> 状态：通过  
> 范围：v3 内部运维、权限与恢复边界、旧 UI 退出、架构门禁

## 1. 结论

本阶段完成的不是第二套“后台皮肤”，而是把运维能力迁移到与消费者相同的 v3 事实源，并删除已经失去业务职责的旧功能台。`index.html` 和 `internal.html` 现在分别承担消费者交易与管理员运维，两个入口拥有独立启动脚本和 UI 组件树；v1/v2 代码只在迁移、验证和历史订单重建链上保留。

生产入口的静态依赖审计结果为 69/69 个 JavaScript 文件全部可达，无缺失导入、无孤儿源码。旧 Canvas、热图、购前评分、实时模拟、规则聊天、跨页结算和 v2 UI composition root 已删除。

## 2. Before / After / Why

| Before | After | Why |
| --- | --- | --- |
| `legacy.html` 同时承载算法演示、模拟、备份、用户和库存 | `internal.html` 只展示场次库存、有效锁座、订单、脱敏用户和恢复边界 | 运维界面应围绕可执行运营事实，而不是历史功能清单 |
| 内部工具使用 v2 状态，消费者使用 v3 状态 | 两个入口共享 v3 repository 与商业应用服务 | 消除双事实源，避免后台看到与真实交易不同的库存 |
| 普通页面可直接打开旧后台功能 | 运维入口有管理员权限门，访客与会员均无法读取数据 | 页面不可发现不等于授权；权限必须在应用服务再次验证 |
| 释放库存、完整导出和数据导入缺少统一风险等级 | 人工释放、含凭证备份和恢复导入均给出具体后果并二次确认 | 高影响操作要让操作者在提交前理解对象、范围和不可逆后果 |
| 脱敏导出与恢复文件语义混杂 | 诊断快照不可恢复；完整备份显式包含本地演示凭证；导入前自动留回滚快照 | 安全分享与灾难恢复是不同产品任务，不能复用含糊文件格式 |
| 旧页面、Canvas controller 和专属测试继续占据架构 | 删除 10k+ 行不可达实现，保留 82 项产品/迁移契约并新增架构防回归 | 测试数量不应成为保留死代码的理由；应验证当前产品和兼容承诺 |

## 3. 运维交互审查

- 访问门在 1440×1000 与 390×844 均无页面级横向溢出，首个管理员用户名输入框自动获得焦点；
- 桌面使用高密度库存表与并列事件面板；手机将指标与恢复卡片改为单列，宽表格只在自身容器内滚动；
- 所有通用按钮和 Dialog 关闭键至少 44px，hover 只对精确指针启用；
- 高频列表和数字刷新没有装饰性动画；按钮按下为 140ms `scale(0.97)`，Dialog 只使用 160ms opacity/scale，并在 reduced-motion 下移除 transform；
- Dialog 打开时焦点落在安全的“返回”操作，背景点击不能提交或关闭危险操作，Escape 与显式关闭键可退出；
- 用户字段只通过 `textContent` 渲染，运维视图模型不包含 credential。

保留的 v2 validator、迁移器、冻结 fixture、历史订单与库存模型均被 v1→v2→v3 启动链引用；它们不是第二套产品实现。下一阶段转向多营业日、多电影和多影院目录，随后再接入支付/退款端口、二维码与真实辅助技术验收。
